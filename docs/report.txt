[1] <Token Refresh Field Name Mismatch>
Expected: POST /api/auth/refresh returns { success, accessToken, refreshToken }.
         Frontend should destructure `accessToken` from the response.
Current: Frontend `refreshAccessToken()` in apiClient.ts was destructuring
         `response.data.token` — but `apiClient.post` returns raw JSON (no
         `.data` wrapper), and the field is `accessToken`, not `token`.
         This caused every 401 retry to fail with "Token refresh failed".
Status: [#] Fixed — apiClient.ts now destructures `accessToken` directly from
        response; RefreshTokenResponse type updated to match backend.

[2] <Customer Routes Not Mounted>
Expected: Customer endpoints (GET /api/customers/orders, GET /api/customers/profile,
         PUT /api/customers/profile, POST /api/customers/register) should be
         accessible at /api/customers/*.
Current: customer.routes.ts existed but was never imported or mounted in app.ts.
         All customer API calls returned 404, which cascaded to 401 → token
         refresh failures on the frontend.
Status: [#] Fixed — customerRoutes imported and mounted at /api/customers in app.ts.

[3] <Cart Routes Not Mounted>
Expected: Cart endpoints (GET /api/cart, POST /api/cart/items, PATCH/DELETE
         /api/cart/items/:itemId, POST /api/cart/checkout) should be accessible
         at /api/cart/*.
Current: cart.routes.ts existed but was never imported or mounted in app.ts.
         Add-to-cart and checkout calls returned 404.
Status: [#] Fixed — cartRoutes imported and mounted at /api/cart in app.ts.

[4] <User Orders Page Calling Seller API>
Expected: Customer users should fetch their orders from a customer-specific
         endpoint (GET /api/customers/orders).
Current: The user orders page (frontend) was calling `listSellerOrders()` which
         hits GET /api/orders — a seller-protected endpoint requiring
         `requireSellerRole("owner", "manager", "staff")`. Customer users get
         401, triggering token refresh, which also failed (see issue #1).
Status: [#] Fixed — Created customer.api.ts with `listMyOrders()` calling
        /api/customers/orders; user orders page updated to use it.

[5] <Product Detail Page Calling Seller API>
Expected: Customer users should view product details via a public endpoint
         that doesn't require seller authentication or permissions.
Current: The user product detail page was calling `getProduct()` which hits
         GET /api/products/:productId — a seller-protected endpoint requiring
         `protect`, `resolveTenant`, and `requirePermission(PRODUCTS_VIEW)`.
         Customer users get 401 → token refresh failure.
Status: [#] Fixed — Added public route GET /api/products/:productId/detail with
        `publicLimiter` only; created `getProductPublic()` in frontend; updated
        user product detail page to use it.

[6] <Cart Service — skuId Null in Compound Unique Lookup>
Expected: Adding a product to cart without a specific SKU should work — the
         cart service should find an existing cart item with `skuId: null`.
Current: `db.cartItem.findUnique()` used the compound unique key
         `cartId_productId_skuId` with `skuId: data.skuId ?? null`. Prisma
         requires all fields in a compound unique to be non-null, so passing
         `null`/`undefined` threw: "Argument `skuId` must not be null."
Status: [#] Fixed — When skuId is provided, use `findUnique` with compound key;
        when skuId is absent, use `findFirst` with `skuId: null` filter.

[7] <Product Search Response Shape Mismatch>
Expected: Frontend `searchProducts()` returns `PaginatedResponse<Product>` with
         `res.data` as a flat array of products.
Current: Backend `searchProducts` returns `{ success, data: { products: [...],
         pagination: {...} } }`. Frontend was doing `setProducts(res.data ?? [])`
         which set `products` to the object `{ products, pagination }` instead
         of the array, causing `products.map is not a function`.
Status: [#] Fixed — Frontend now extracts `res.data.products` instead of `res.data`.

[8] <Rate Limiter Misconfiguration on Customer Routes>
Expected: Customer-facing routes (profile, orders) should use `publicLimiter`
         or a customer-specific rate limiter, not `sellerLimiter`.
Current: customer.routes.ts uses `sellerLimiter` on GET /profile, PUT /profile,
         and GET /orders. The `sellerLimiter` may have stricter limits intended
         for seller endpoints, potentially rate-limiting customers prematurely.
Status: [ ] Open — Consider switching to `publicLimiter` or a dedicated

[9] <Cart Uses sellerLimiter Instead of publicLimiter>

Current: cart.routes.ts uses `sellerLimiter` on all endpoints. This applies

Status: [ ] Open — Consider switching to `publicLimiter` for cart endpoints.

ARCHITECTURAL ISSUE — MULTI-SELLER ORDERS
================================================================================

[10] <Single-Seller Cart Constraint — Cannot Order From Multiple Sellers>

Scenario:
  A customer wants to order Product A (from Seller 1) and Product B
  (from Seller 2) in a single checkout.

Expected:
  The customer should be able to add products from different sellers to their
  cart and checkout. The system should either:
    (a) Split the cart into separate orders — one per seller, each with its
        own order ID, status tracking, fulfillment, and payouts; OR
    (b) Create a single parent order with child sub-orders per seller, allowing
        unified payment but independent fulfillment per seller.

Current:
  The cart enforces a SINGLE seller per cart. When a customer adds a product
  from a different seller:

    cart.service.ts line 79-84:
      if (cart.sellerId && cart.sellerId !== product.sellerId) {
          await db.cartItem.deleteMany({ where: { cartId: cart.id } });
          await db.cart.update({
              where: { id: cart.id },
              data: { sellerId: product.sellerId },
          });
      }

  This DELETES all existing cart items and replaces the sellerId with the new
  seller. The customer's previous cart contents are silently destroyed.

  At checkout (cart.service.ts line 180-182):
      const { cart, items, subtotal, sellerId } = await getCartWithItems(userId);
      if (!items.length) throw new Error("Cart is empty");
      if (!sellerId) throw new Error("Cart has no seller assigned");

  Only one `sellerId` is used — the entire cart is sent to `orderService.createOrder()`
  with a single `sellerId`, creating one Order row.

  The Order model itself has a single `sellerId` field:
      prisma schema:
        model Order {
          sellerId  String
          seller    Seller @relation(fields: [sellerId], references: [id])
          ...
        }

  This means:
    - One order = one seller (enforced at DB schema level)
    - No support for multi-seller orders or order splitting
    - Customer loses cart items when switching sellers
    - No "parent order" or "order group" concept exists

  Impact:
    - Customer cannot buy from multiple sellers in one checkout
    - Customer cannot see a combined cart with items from different sellers
    - No unified payment for multi-seller purchases
    - Each seller transaction must be a separate checkout

  Proposed Solutions:

  Option A — Cart Splitting at Checkout (Recommended, minimal schema change):
    1. Remove the single-seller enforcement in `addItem()` — allow items from
       multiple sellers in the same cart.
    2. Remove `sellerId` from the Cart model (or make it optional/unused).
    3. At checkout, group cart items by `product.sellerId` and create one
       Order per seller group in a single transaction.
    4. Return an array of created orders to the frontend.
    5. Frontend shows "Order placed — N orders created from your cart."
    6. Each order has independent status, fulfillment, shipment, and payout.
    7. Payment: Either charge separately per order, or charge the total and
       split internally (requires payment gateway split-payment support or
       manual reconciliation).

  Option B — Parent Order with Sub-Orders (Larger schema change):
    1. Add a `ParentOrder` model with `parentId` on `Order`.
    2. Create one parent order for payment, then child orders per seller.
    3. Payment is unified at parent level; fulfillment is per child order.
    4. Requires schema migration, new API endpoints, and UI changes.
    5. Better for customer experience (single payment, single receipt) but
       more complex to implement.

  Recommendation:
    Option A is simpler and works within the existing schema. The Order model
    already has a single sellerId — just create multiple Order rows. The main
    changes needed:
      - cart.service.ts: Remove single-seller enforcement in addItem()
      - cart.service.ts checkout(): Group items by seller, create N orders
      - Frontend: Handle array of orders in checkout response
      - Frontend: Show cart items grouped by seller with per-seller subtotal

Status: [ ] Open — Architectural decision needed.
